Section 01 · Minor Details
Hardware Components

Physical devices that form the foundation of the system.

Importance Level 1/5
Compute Unit

NVIDIA Jetson Nano

Runs all ROS2 nodes. Connected to internet for IoT publishing. Outputs PWM signals through Low Pass Filters to the PLC.

Vision Sensor

RGB-D Camera

Attached to Jetson Nano. Provides both RGB (color) and Depth streams. Used for pear inspection and speed estimation.

Motor Controller

PLC

Receives 0–3.3 V analog voltage (converted from PWM by a Low Pass Filter) to control conveyor and servo motor speeds.

Conveyor 1 (Conv1)

Belt- 175 cm

Left/incoming belt. Speed measured by counting encoder pulses (line crossings), then multiplying by belt length (175 cm).

Conveyor 2 (Conv2)

Belt- 115 cm

Right/outgoing belt. Same speed measurement method as Conv1 but length is 115 cm.

Physical Actuator

Servo / Sorter

Activated by A1–B3 Action nodes to physically move (reject or pass) each pear based on vision analysis result.

Section 05 · Minor Details
Speed Conversion Formulas (m/s → Voltage → RPM)

The Jetson Nano outputs a linear voltage (LV) that maps to a physical speed. The PLC then converts this to motor RPM.

Importance Level 2/5
/* Speed mapping table */ m/s → Voltage (V) → RPM (0–75) 0.0 → 0.1 V → 0 RPM 0.5 → 3.3 V → 75 RPM /* When LV = 0 */ V = 0.1 V → R = 0 RPM /* Conv relationship */ Conv₂ = LV Conv₁ = 3.3 − Conv₂ // always sum to 3.3
Key Rule

Conv1 + Conv2 = 3.3 V (always)

The two conveyor speeds are complementary. When Conv2 is fast (busy), Conv1 slows down- and vice versa. This balances the pear flow between belts.

Idle State

Minimum voltage = 0.1 V

Even at 0 m/s, the system sends 0.1 V (not 0 V) to prevent the PLC from interpreting a dead signal as a fault. True zero speed = RPM 0 but V = 0.1.

Section 06 · Minor Details
PWM → Voltage Conversion (Low Pass Filter)

The Jetson Nano cannot output true analog voltages directly. It outputs PWM (Pulse Width Modulation) signals that are smoothed into analog voltages by hardware Low Pass Filters before reaching the PLC.

Importance Level 2/5
Jetson Nano
→
PWM Signal
→
Low Pass Filter
→
Analog Voltage (0–3.3 V)
→
PLC
LF for Conv1

Low Pass Filter- "Conv1"

Smooths the PWM output for Conveyor 1. Converts digital duty cycle into a clean 0–3.3 V analog signal.

LF for Conv2

Low Pass Filter- "Conv2"

Smooths the PWM output for Conveyor 2. Same circuit, separate channel for the second belt.

Speed publisher example: When Reference = 37.5 RPM → V = 1.65 V → LV = 0.25. The error voltage is sent as PWM → LPF → PLC for both conveyors.
Section 07 · Supporting Details
Conveyor Speed Measurement Methods

Conv1 and Conv2 compute their belt speeds independently using encoder/line-counting logic. Servo speed is measured directly from system feedback.

Importance Level 3/5
Conv1_Speed Node

Belt 1- 175 cm

  1. Count how many turns "a placed line at the conveyor" makes
  2. Get the time for each full cycle
  3. Multiply cycle count × 175 cm = distance
  4. Distance ÷ time = speed
Conv2_Speed Node

Belt 2- 115 cm

  1. Same counting method as Conv1
  2. Get the time for each full cycle
  3. Multiply cycle count × 115 cm = distance
  4. Distance ÷ time = speed
Servo_Speed Node

Direct Measurement

Measures the actual system speed directly from the servo motor feedback. Used as the real-time feedback signal into the PID speed controller.

Section 08 · Core Algorithm
Vision Algorithm- 9-Step Pipeline

Classical computer vision processing chain applied to each camera frame to detect infections on pears, extract location and color, and segment the pear from the background.

Importance Level 4/5
Raw Frame
→
9-Step Pipeline
→
Clean Pear + Infection
→
Infection Mask Only
  • 1
    CLAHE- Adaptive Histogram Equalization Contrast Limited Adaptive Histogram Equalization. Enhances local contrast in different regions of the image independently, making subtle infection patterns more visible even in non-uniform lighting conditions on the belt.
  • 2
    Bilateral Filter Edge-preserving noise smoothing filter. Removes sensor noise and fine texture without blurring the sharp boundaries between healthy and infected pear tissue. Unlike Gaussian blur, it keeps object edges sharp.
  • 3
    Color Space Conversion Converts the frame from BGR (camera native) to an appropriate color space (likely HSV or LAB) to make color-based segmentation robust to brightness variation on the conveyor.
  • 4
    Find the Color Range Determines the HSV or color range thresholds for both healthy pear skin and infected tissue. This step isolates the specific RGB/HSV color range that corresponds to pear infections (brown spots, discoloration).
  • 5
    Bitwise AND (Mask Application) Applies the color mask to the image using a bitwise AND operation. Only pixels that fall within the defined color range are kept; everything else is zeroed out, isolating the candidate infection regions.
  • 6
    Morphological Cleanup- Opening & Closing Opening (erosion → dilation) removes small noise blobs. Closing (dilation → erosion) fills small holes inside infection regions. Together they produce clean, solid infection masks without false positives.
  • 7
    Otsu's Thresholding Automatic global thresholding that finds the optimal binary cutoff value by analyzing the image histogram. Converts the grayscale mask into a clean binary (black/white) image. No manual threshold tuning required.
  • 8
    Contour Analysis Finds and analyzes the outer boundaries of all white regions in the binary mask. Each contour corresponds to one infection region. Used to extract: bounding rectangle (location), contour area (infection area), and centroid coordinates.
  • 9
    Outputs Final results published to downstream nodes: infection area (pixels²), infection location (x, y centroid), infection RGB color code, binary status (infected / clean), and action trigger for the actuator node.
Background removal: The pipeline also produces a clean "pear-only" image by removing the conveyor belt background- giving downstream nodes a clean pear silhouette for accurate area and volume estimation.
Section 09 · Core Algorithm
Speed Algorithm- PID Control Loop

A PID (Proportional-Integral-Derivative) controller manages each conveyor and the servo motor speed. Three parallel PID loops maintain precise speed control.

Importance Level 4/5
/* PID Control Structure */ Frames ──► Model ──►[Σ]──► Controller "PID" ──► Conv1_model ──► S₁ ──► RPM ▲ | Reference ──────────┘ Model ◄──── Conv1_Frame ◄──────────────────┘ [Σ] ──► PID ──► Conv2_model ──► S₂ ▲ | └──── Model ◄── Conv2_Frame ◄───┘ [Σ] ──► PID ──► Servo_model ──► SV ▲ | └──── Model ◄── from_Servo ◄────┘
PID Loop 1

Conv1 Speed

Controls Conveyor 1 speed. Setpoint from Reference speed. Feedback from Conv1_Frame pixel analysis. Output → S₁ (RPM command)

PID Loop 2

Conv2 Speed

Controls Conveyor 2 speed. Complementary to Conv1 (Conv1 + Conv2 = 3.3 V). Feedback from Conv2_Frame. Output → S₂ (RPM command)

PID Loop 3

Servo Speed

Controls the servo/ejection mechanism. Feedback from servo encoder. Output → SV (servo voltage command). Also receives reference speed signal.

Speed publisher logic: Takes current speed from servo_speed → compares with Reference → computes error → sends Conv1 as PWM (error voltage). When Conv1 = error: Conv2 = 3.3 − Conv1.
Section 10 · Core Algorithm
Reference Speed Algorithm

Computes the optimal belt reference speed dynamically by tracking pear centroids across the vision zone. The result feeds into the PID controller as the setpoint.

Importance Level 4/5
  • 1
    Find the pears Detect all pears currently in the vision zone frame using the vision pipeline output.
  • 2
    Check pear count If more than 6 pears detected simultaneously → system is at maximum capacity: set Conv1 → 0, Conv2 → 3.3 (full throughput mode). This prevents belt overloading.
  • 3
    Find centroids For each pear, compute its centroid (geometric center x,y pixel coordinate) from the contour analysis output.
  • 4
    Track centroid over time Record centroid position at timestep t₁ and again at t₂. Store {time, location} pairs to compute displacement over time.
  • 5
    Compute per-pear speed Speed = pixel displacement ÷ elapsed time. Convert pixels/second to m/s using the known camera-to-belt mapping. One speed value per pear per row.
  • 6
    Average speed (if pears only moving) Average Speed = (S₁ + S₂ + … + Sₙ) / n. Only moving pears are included. Stationary pears (e.g., jamming) are excluded from the average.
  • 7
    Publish speed Publishes the computed reference speed to the /main_speed topic. This becomes the setpoint for all three PID loops.
Belt EMPTY state

Conv1 → 3.3 V · Conv2 → 0 V

When no pears detected: Conv1 runs at full speed to pull in new pears, Conv2 is idle.

Belt CROWDED state (>6 pears)

Conv1 → 0 V · Conv2 → 3.3 V

When belt is over-full: Conv1 stops accepting more, Conv2 runs at full to clear the queue.

Section 11 · Node Reference
All ROS2 Nodes- Descriptions

Complete catalogue of every node in the system, what it subscribes to, what it publishes, and what it does internally.

Importance Level 4/5
Node · Capture Layer

frame_capture

The entry point. Receives raw frames directly from the camera. Distributes frames to three downstream divider nodes: frame_divider, frame_speed_divider, and volume_divider. Acts as the main frame broadcaster- every other node depends on it.

Node · Frame Routing

frame_divider

Splits the incoming frame into 6 sub-frames (A1–B3) and sends each to its corresponding analysis node. Also outputs Conv1, Conv2, and Servo speed sub-frames.

Node · Frame Routing

frame_speed_divider

Extracts Conv1 (cols 0–256), Vision (cols 257–1024), and Conv2 (cols 1025–1280) sub-frames for the speed computation pipeline.

Node · Volume/Mass

volume_divider

Routes depth frames to the area estimation nodes (A1…B3 Area nodes) for 3D volume and mass computation per pear.

Node · Initialization

classical_vision_initialization

Runs once at system startup. Loads and broadcasts all system parameters (color thresholds, calibration values, filter settings) to the 6 vision nodes. Acts as the configuration hub.

Nodes · Vision Analysis × 6

A1_vision A2_vision A3_vision B1_vision B2_vision B3_vision

Six parallel nodes, one per belt zone. Each runs the 9-step vision algorithm on its sub-frame. Outputs: infection detection result (location, area, color, status, action flag) published to the corresponding action and data-collector nodes.

Nodes · Physical Actuation × 6

A1_action through B3_action

Each action node:
1. Gets the detection result from its paired vision node.
2. Commands the physical actuator (ejector/sorter) to accept or reject the pear.
Action nodes are the only nodes that directly trigger physical hardware movement.

Nodes · Data Collection × 6

A1_data_collector through B3_data_collector

One per zone. Performs 6 tasks:
1. Generate unique Pear ID (e.g., A1_00001)
2. Retrieve infection color
3. Retrieve infection area
4. Retrieve infection location
5. Compute pear surface area
6. Send all data to the database node

Node · Infection Summary

infection_description

Aggregates infection data from all zones. Extracts: area of infection, (x,y) location on pear, RGB color code of infection. Assigns a system-wide unique ID. Outputs to two destinations: publishes to IoT system + saves to local database.

Node · Belt Speed

conv1_speed

Counts belt encoder pulses. Multiplies by 175 cm. Publishes Conv1 actual speed to speed pipeline.

Node · Belt Speed

conv2_speed

Same method as Conv1. Multiplies by 115 cm. Publishes Conv2 actual speed to speed pipeline.

Node · Servo Speed

servo_speed

Reads servo encoder directly. Publishes real-time servo speed as the primary feedback for the PID controller.

Node · Reference Speed

main_speed → centroid_time_speed

Detects pear centroids, tracks them across frames, computes per-pear velocity, averages all moving pears, and publishes the reference speed. This is the setpoint for the PID controller. Also sends servo feedback to the servo PID loop.

Node · Speed Publisher (to PLC)

speed_publisher

The final speed node. Collects Conv1, Conv2, and Servo speeds → runs PID error computation → converts to PWM → sends through Low Pass Filters → outputs 0–3.3 V to the PLC for all three motors.

Nodes · Area × 6

A1_area_node…B3_area_node

Compute the 2D surface area of each pear from the segmented silhouette image. Published to volume and IoT topics.

Node · Volume

volume_estimation

Uses depth data + 2D area to estimate the 3D volume of each pear. Publishes to IoT.

Node · Mass

mass_estimation

Estimates pear mass from volume (using density model). Publishes to IoT and database.

Node · Persistence

data_collection_node (Database)

Receives data from all 6 data-collector nodes. Processes A1 first, then A2…B3, then repeats. For each: assigns ID, writes all fields, moves to next zone. Also publishes to the database web page.

Node · IoT Web Publisher

webpage_publisher_status

Publishes live system status to the IoT overview web page. Broadcasts: motor status array, accepted/rejected counts, and all per-pear fields listed in the IoT schema.

Section 17 · MOST IMPORTANT
Critical System Requirements & Notes

These constraints define the fundamental operational boundaries of the entire system. Missing any one of them will cause system failure.

Importance Level 5/5- READ FIRST
⚠

Non-Negotiable System Constraints

  • 1-Second Action Cycle- VERY VERY IMPORTANT.  The system must take a complete vision + detection + action decision every 1 second. This is the fundamental timing constraint for belt speed, node scheduling, PID update rate, and actuator response. Every node must complete its processing within this window. Missing this cycle means a pear passes through without inspection.
  • PWM-to-Voltage via Low Pass Filter- mandatory hardware.  The Jetson Nano outputs digital PWM; the PLC requires smooth analog 0–3.3 V. Without the Low Pass Filters on both Conv1 and Conv2 channels, the PLC will receive a pulsing digital signal and behave erratically or fail.
  • Conv1 + Conv2 speeds must always sum to 3.3 V.  The dual-conveyor balance rule is a system invariant. Violating it by independently setting both speeds will overload one side or starve the other, causing pear pile-ups or gaps.
  • Maximum 6 pears in the vision zone simultaneously.  If the reference speed algorithm detects >6 pears, it automatically switches to overflow mode (Conv1 → 0, Conv2 → 3.3). This is a hard capacity limit that must not be bypassed by adjusting belt speeds manually.
  • IoT data persists offline and syncs on reconnect.  Infection samples and status data are saved locally at all times. After a system shutdown and restart with internet access, all locally buffered data is uploaded to the cloud. This means no inspection data is lost during offline periods.
  • classical_vision_initialization must run before any vision nodes.  All 6 vision nodes depend on system parameters published by this node. Starting vision nodes before initialization completes will cause them to run with undefined parameters.
  • Database writes are sequential (A1 → A2 → … → B3 → repeat).  The data_collection_node processes zones in strict order. Any disruption to this sequence causes ID assignment errors (duplicate IDs or gaps).
